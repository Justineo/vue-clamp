# Vue Clamp 1.4.0 to 1.5.0 performance analysis

## Data source

This note is based on the package benchmark matrix generated from `/tmp/vue-clamp-matrix-2s-cv`
and stored locally as `journey/research/310-package-benchmark-matrix.local.json`.

The benchmark compares published package versions through the public component API. The report
sampling mode uses at least 2 seconds of measured wall time per scenario, a minimum of 3 measured
samples, median active time as the primary timing value, and active-time CV as the low-confidence
marker.

## Aggregate result

| Component     | 1.4.0 active | 1.5.0 active |  Delta | Rect reads | Item slot calls | Interpretation                                                              |
| ------------- | -----------: | -----------: | -----: | ---------: | --------------: | --------------------------------------------------------------------------- |
| LineClamp     |    2094.1 ms |    2225.7 ms |  +6.3% |       0.0% |             N/A | Timing-only regression in the same measured work shape.                     |
| InlineClamp   |    1796.6 ms |    1675.3 ms |  -6.7% |       0.0% |             N/A | Runtime/render-path improvement, not fewer DOM reads.                       |
| RichLineClamp |    2168.4 ms |    2185.5 ms |  +0.8% |       0.0% |             N/A | Mostly neutral; small timing-only movement.                                 |
| WrapClamp     |    8026.3 ms |    3886.3 ms | -51.6% |     -60.4% |          -70.0% | Dominant improvement from less DOM probing and fewer item slot invocations. |

Across all comparable scenarios, 1.5.0 improves active time by 29.2%, settled time by 9.1%, rect
reads by 58.0%, item slot calls by 70.0%, and long-task count by 17.5%.

## Main improvements

The 1.5.0 improvement is dominated by WrapClamp. The largest scenario wins are all grow-heavy
wrapped-item cases:

| Scenario                                | Active delta | Rect reads | Item slot calls |
| --------------------------------------- | -----------: | ---------: | --------------: |
| `wrap-no-affix-tiny-item-wide-grow`     |       -80.0% |     -87.4% |          -92.5% |
| `wrap-no-affix-heavy-item-grow`         |       -66.8% |     -67.5% |          -72.2% |
| `wrap-no-affix-hidden-grow`             |       -65.9% |     -67.5% |          -72.2% |
| `wrap-static-before-dynamic-after-grow` |       -62.8% |     -71.8% |          -63.6% |
| `wrap-static-after-grow`                |       -63.3% |     -77.5% |          -68.0% |
| `wrap-before-affix-grow`                |       -55.9% |     -43.4% |          -78.6% |

The source-level reason is the WrapClamp solver rewrite in 1.5.0:

- It records item widths during real measurements and reuses those widths for later estimates.
- It can estimate no-affix / static-flow counts without rendering every candidate item.
- Grow paths materialize a bounded frontier and then toggle existing item shells during search
  instead of repeatedly re-rendering each candidate prefix.
- The fallback linear settlement still exists, but it is reached after cheaper proof/hint paths.

The item slot call reduction is especially important because the benchmark intentionally renders
many WrapClamp rows with public `item` slots. In 1.4.0, grow paths repeatedly invoked item slots for
candidate prefixes. In 1.5.0, the renderer only renders up to the visible count or bounded
materialized count, so heavy item content no longer scales with every probe.

InlineClamp also improves in continuous width-sweep scenarios:

| Scenario                              | Active delta | Work counters |
| ------------------------------------- | -----------: | ------------- |
| `inline-path-middle-batch-continuous` |       -17.3% | unchanged     |
| `inline-path-end-batch-continuous`    |       -17.0% | unchanged     |

Because DOM-read counters are unchanged, this is best attributed to component-shell/render-path
changes from the SFC migration and local state simplification, not to a changed text-clamp search
algorithm.

## Main regressions

The largest timing regressions are:

| Scenario                               | Active delta | Absolute delta | Work counters | Confidence      |
| -------------------------------------- | -----------: | -------------: | ------------- | --------------- |
| `line-summary-batch-continuous`        |       +44.6% |       +50.9 ms | unchanged     | CV 4.2% -> 0.7% |
| `line-cta-affix-batch-continuous`      |        +9.0% |       +46.8 ms | unchanged     | CV 2.5% -> 2.3% |
| `rich-metadata-affix-batch-continuous` |        +4.6% |       +24.5 ms | unchanged     | CV 1.3% -> 1.9% |
| `inline-path-middle-batch-jitter`      |        +5.2% |       +23.2 ms | unchanged     | CV 7.5% -> 1.6% |
| `rich-metadata-affix-batch-jitter`     |        +3.1% |       +19.2 ms | unchanged     | CV 0.3% -> 1.4% |
| `line-prefixed-summary-batch-jumps`    |       +25.5% |       +11.5 ms | unchanged     | CV 1.8% -> 7.6% |
| `inline-split-file-path-batch-jumps`   |       +13.9% |        +8.2 ms | unchanged     | CV 5.1% -> 3.4% |

For LineClamp, the most important regression is `line-summary-batch-continuous`: it is a native
eligible multiline default-ellipsis row with no affix. The native implementation path and measured
DOM-read counts did not change between 1.4.0 and 1.5.0. The source-level difference is the component
shell: 1.5.0 moved LineClamp from a hand-written render function to an SFC/template shell and
`defineModel`-based controlled state. A concrete likely contributor is that the SFC template renders
the reusable `MultilineAffixSlot` functional component for both before and after positions even when
there is no slot content. In 1.4.0 the hand-written render function skipped those wrappers when slot
content was absent.

For the affix LineClamp and RichLineClamp rows, rect reads, resize callbacks, mutation records, and
component instance counts are unchanged. These regressions are therefore not evidence of extra
clamp probes. They are timing-only overhead in the same externally observable workload, most likely
from the SFC shell, the reusable affix-slot functional component path, and reactive style/render
boundaries.

Small WrapClamp regressions such as `wrap-table-demo-width-sweep` (+3.4%) and
`wrap-after-affix-shrink` (+2.6%) also have unchanged rect and slot counters. They are not the main
story and should not be treated as solver regressions unless a profiler shows new hot frames.

## Bottom line

1.5.0 is a clear WrapClamp performance release. The solver rewrite cuts the expensive work that
actually matters for wrapped-item scenarios: DOM rect probing and public item-slot invocation.

The regressions are real in the timing data, especially `line-summary-batch-continuous`, but they
are not caused by extra DOM measurement. They are component-shell/runtime overhead introduced while
migrating to SFCs and simplifying public typing. If those regressions matter enough to optimize, the
next diagnostic step should be an isolated commit-level run or browser profile focused on the
native LineClamp SFC path and affix slot render path.

## DefineRender follow-up experiment

After this analysis, `LineClamp`, `RichLineClamp`, and `InlineClamp` were migrated from authored
templates to setup-local render functions bound with `defineRender(render)`. `WrapClamp` was already
render-only. The goal was to test whether avoiding template-emitted `MultilineAffixSlot` component
calls would recover 1.5.0 timing regressions.

The package benchmark was rerun against the current workspace and published `vue-clamp@1.5.0` using
the same public matrix and report sampling mode:

| Component     | Published 1.5.0 | Current defineRender | Delta |
| ------------- | --------------: | -------------------: | ----: |
| All           |       9903.3 ms |            9975.5 ms | +0.7% |
| LineClamp     |       2230.9 ms |            2160.1 ms | -3.2% |
| InlineClamp   |       1715.6 ms |            1819.0 ms | +6.0% |
| RichLineClamp |       2162.0 ms |            2147.9 ms | -0.6% |
| WrapClamp     |       3794.8 ms |            3848.4 ms | +1.4% |

The intended Line/Rich recovery did show up:

| Scenario                               | Published 1.5.0 | Current defineRender |  Delta |
| -------------------------------------- | --------------: | -------------------: | -----: |
| `line-summary-batch-continuous`        |        166.3 ms |             142.0 ms | -14.6% |
| `line-cta-affix-batch-continuous`      |        566.8 ms |             544.4 ms |  -4.0% |
| `line-cta-affix-batch-jitter`          |        844.3 ms |             811.9 ms |  -3.8% |
| `rich-metadata-affix-batch-continuous` |        551.4 ms |             536.5 ms |  -2.7% |
| `rich-metadata-affix-batch-jitter`     |        640.4 ms |             627.8 ms |  -2.0% |

But migrating InlineClamp to a manual render function was not a win in this run:

| Scenario                              | Published 1.5.0 | Current defineRender |  Delta |
| ------------------------------------- | --------------: | -------------------: | -----: |
| `inline-path-middle-batch-jitter`     |        418.8 ms |             456.8 ms |  +9.1% |
| `inline-path-middle-batch-continuous` |        416.9 ms |             435.8 ms |  +4.5% |
| `inline-split-file-path-batch-jumps`  |         39.7 ms |              60.4 ms | +52.1% |

Counters were effectively unchanged in the defineRender comparison, so the movement is still
render/runtime timing rather than changed clamp probes. The practical takeaway is that render-only
SFCs are promising for LineClamp and mildly helpful for RichLineClamp, but should not automatically
be assumed faster for InlineClamp. If optimizing purely for current benchmark totals, keep the
LineClamp/RichLineClamp render-only migration and reconsider InlineClamp separately.

The deeper reason is that the experiment changes two different costs in opposite directions:

- `LineClamp` and `RichLineClamp` templates in published 1.5.0 always instantiate the reusable
  `MultilineAffixSlot` VNode path for both before and after positions. In no-affix rows this is
  pure shell work; in affix rows it adds a component boundary and prop patching around slot
  rendering. The render-only experiment removes or inlines that boundary, which recovers the
  targeted Line/Rich regressions.
- Authored templates also receive Vue compiler output: block tracking, patch flags, hoisted dynamic
  prop key arrays, and generated template-ref handling. Handwritten `h()` render functions do not
  get those compiler optimizations automatically. `InlineClamp` had no `MultilineAffixSlot` boundary
  to remove, so the experiment mostly traded a compiler-optimized template for fresh hand-built
  VNodes, props objects, children arrays, and function refs. That is why InlineClamp regressed even
  though its DOM-read and mutation counters were unchanged.
- The LineClamp recovery is partial because the 1.4.0 render path was not just render-only; it was a
  leaner hand-written `defineComponent` path. The current SFC path still carries `defineModel` /
  `useModel`, template-oriented computed boundaries such as `rootStyle`, `contentStyle`,
  `nativeMode`, `bodyStyle`, and `renderedText`, plus function refs for DOM nodes. The experiment
  removed the biggest avoidable template component boundary, but it did not recreate the entire
  1.4.0 runtime shape.

Most of the measured movement is in `updateMs`, not `settleWaitMs`. Counters for layout reads,
resize callbacks, and mutation records are nearly unchanged, so these deltas should be interpreted
as Vue render/patch/ref/slot overhead rather than browser layout or clamp-search work.

## Hybrid SFC render optimization pass

The follow-up implementation kept the public SFC shell constraint but stopped treating render-only
as universally faster:

- `InlineClamp` returned to an authored template with `useTemplateRef` because its runtime structure
  is static and the template compiler's block/patch/ref output consistently fits that shape better
  than a handwritten `h()` render function.
- `LineClamp` and `RichLineClamp` stayed render-only, but root/content/body/text/probe refs now use
  direct shallow ref objects for stable element refs. Function refs remain only for affix wrappers,
  where appearance/disappearance must schedule a reclamp after Vue commits slot DOM.
- One-use render aliases such as native mode, body style, rendered text, and slot props were moved
  out of `computed` effects and into render-local values, keeping `computed` for preparation work
  that is actually cached across reclamps.
- `LineClamp` also reuses stable style objects on the common no-`maxHeight` path and avoids rendering
  a null source-text placeholder when the visible text is not accessibility-rewritten.

The first full matrix after the hybrid pass showed the expected direction versus the previous
all-`defineRender` workspace run:

| ------------- | -----------------------: | ----------: | -----: |
| All           |                9975.5 ms |   9091.9 ms |  -8.9% |
| LineClamp     |                2160.1 ms |   2071.6 ms |  -4.1% |
| InlineClamp   |                1819.0 ms |   1360.0 ms | -25.2% |
| RichLineClamp |                2147.9 ms |   1911.3 ms | -11.0% |
| WrapClamp     |                3848.4 ms |   3749.0 ms |  -2.6% |

Two repeated runs of the final code showed meaningful cross-run drift, especially in long
Inline/Rich continuous scenarios, so the conservative reading uses their average rather than a
single best run:

one browser session to reduce drift.

## Line and Rich focused probe reuse pass

A later LineClamp/RichLineClamp pass focused on the affix-heavy scenarios that still showed timing
pressure after the hybrid shell work.

The retained LineClamp optimization is deliberately narrow: when the previous measured text result
was clamped, the prepared boundary array is unchanged, and the root width has not grown beyond the
local-search window, LineClamp skips the separate full-source fit probe and lets the warm candidate
search include the full-source candidate. This removes one text mutation and one fit read from
stable clamped resize passes without changing the cold path. Large grows still keep the explicit
full-fit probe, and the search can still rediscover the full source when changed limits make it fit.
The text layout helper now requires that measured root width as part of its input instead of
falling back to its own root bounding-rect read. The package component path already had that width;
source direct Line benchmarks therefore moved their root bounding-rect reads to zero while keeping
client-rect reads unchanged, for example `direct-line-continuous-warm` 560 -> 0 and
`direct-line-jumps-warm` 109 -> 0.

LineClamp's native CSS overflow path now treats `clientWidth <= 0` as the only unmeasurable-width
guard instead of also reading `getBoundingClientRect().width`. Browser correctness tests still pass,
and package smoke moved native Line rows from extra bounding-rect reads to none:
`line-title-single-batch-sweep` 256 -> 0, `line-title-single-affix-batch-sweep` 256 -> 0, and
`line-summary-batch-continuous` 1120 -> 0. The same smoke run moved those rows' median active times
from 51.6 ms to 48.5 ms, 60.1 ms to 58.2 ms, and 215.7 ms to 197.6 ms respectively.

The retained RichLineClamp optimization keeps the hidden probe scaffold mounted across candidates:
the probe root keeps a stable content/body pair, affix clone slots are cached by source HTML plus
measured geometry signature, and the content child list is rebuilt only when affix presence or clone
identity changes. This preserves correctness for dynamic slot sizing while avoiding repeated affix
clone and `replaceChildren` work in metadata affix scenarios.

Rich probes now carry their measured width into `clampRich`. The hidden component probe has just
been assigned that width, and direct benchmark probes already know their current test width, so
`clampRich` can use the snapshot as the zero-width guard instead of reading the probe root's
bounding rect again. Source benchmarks moved `component-rich-continuous` bounding-rect reads from
220 to 110 and `component-rich-jumps` from 48 to 24; after making width part of the required probe
contract, direct Rich source rows also dropped their root bounding-rect reads to zero while keeping
client-rect reads unchanged. Package smoke showed the same component-path structural reduction:
`rich-metadata-affix-batch-continuous` 3520 -> 1760, `rich-metadata-affix-batch-jitter` 3712 ->
1856, and `rich-inline-markup-batch-continuous` 3520 -> 1760, with client-rect reads and mutation
records unchanged.

The source-level Rich clamp pass also replaced boundary path string comparison with direct path
array comparison. This removes repeated `path.join(".")` allocation on hot `sameBoundaryPoint`
checks. In `rich.browser.benchmark.ts`, median total time moved from 36.35 ms to 35.05 ms for
`direct-rich-continuous-warm`, 29.95 ms to 28.00 ms for `direct-rich-jitter-warm`, 41.60 ms to
36.85 ms for `direct-rich-dense-jumps-cold`, and 16.25 ms to 14.65 ms for
`component-rich-continuous`, with DOM-read and clone counters unchanged.

The shared layout-signature helper now builds the signature with a simple loop instead of
`map(...).join("|")`. This is intentionally a small allocation cleanup on the Line/Rich lifecycle
path. In a source benchmark comparison, text warm rows moved slightly positive while Rich rows were
mostly unchanged; DOM-read counters did not change, so this should be treated as a low-risk runtime
cleanup rather than a headline algorithmic win.

A final text-helper cleanup kept only changes that survived repeated source checks. ASCII grapheme
boundary preparation now uses a direct offset loop instead of `Array.from(..., callback)`, and text
fit checks opt into a narrow rect-count shortcut when only `maxLines` is active. Compared with the
post-`offsetWidth` source run, final direct Line rows kept the same DOM-read counts while moving
`direct-line-continuous-cold` from 107.20 ms to 101.00 ms, `direct-line-continuous-warm` from
44.10 ms to 40.70 ms, `direct-line-jitter-cold` from 53.60 ms to 49.80 ms,
`direct-line-jitter-warm` from 35.70 ms to 33.60 ms, `direct-line-jumps-cold` from 20.50 ms to
19.00 ms, and `direct-line-jumps-warm` from 20.70 ms to 19.50 ms. The same ASCII boundary change
also helps Rich preprocessing modestly in direct source rows, for example
`direct-rich-continuous-warm` 30.15 ms -> 29.40 ms and `direct-rich-dense-jumps-cold` 36.60 ms ->
35.60 ms, while component Rich timing stayed within observed drift.

Single package smoke runs after these micro-optimizations showed unchanged structural counters but
large timing drift, including slower native Line rows whose code path had not changed. Treat those
package smoke runs as shape validation only; source benchmarks are the stronger evidence for the
text-helper micro changes.

The remaining max-height-heavy package rows still carried root bounding-rect reads inside every fit
probe. Caching the root's visible top once per clamp pass, while continuing to read current
`clientHeight` for each candidate, preserved browser correctness and reduced those structural
reads in package smoke: `line-height-card-batch-jumps` bounding-rect reads moved 2976 -> 576, and
`rich-height-card-batch-jumps` moved 2715 -> 576. Client-rect reads and mutation records were
unchanged, confirming the search shape stayed the same while each max-height candidate became
cheaper.

RichLineClamp then adopted the same root-width stance as the measured Line path for component probe
sizing: the visible root width is read once and the probe receives that width as an explicit
snapshot. A dedicated fractional-width Rich browser test keeps this guarded. Package smoke showed
the expected structural drop without changing search counters:
`rich-metadata-affix-batch-continuous` bounding-rect reads 1760 -> 0,
`rich-metadata-affix-batch-jitter` 1856 -> 0, `rich-inline-markup-batch-continuous` 1760 -> 0, and
`rich-height-card-batch-jumps` 576 -> 288.
The corresponding active times were mixed but generally favorable in that smoke run, while
client-rect reads and mutation records were unchanged.

Rich preparation also now builds text and fallback boundary points through one small helper instead
of keeping a hand-written loop plus `slice(1).map(...)`. Source timing was mixed within normal
noise, but this removes one avoidable allocation path and makes word fallback construction match the
primary cut construction exactly.

RichLineClamp then gained a guarded analogue of the Line full-fit skip. When the previous probe
result was clamped and the probe width has not grown beyond the warm-search window, the clamp pass
skips the standalone full-rich fit read and lets the coarse search include the full state as its
last candidate. A browser test covers the small-grow case where the full source becomes valid again.
Package smoke showed a structural client-rect reduction in the target component rows:
`rich-metadata-affix-batch-continuous` 8464 -> 6720,
`rich-metadata-affix-batch-jitter` 11424 -> 9568,
`rich-inline-markup-batch-continuous` 8352 -> 6608, and
`rich-height-card-batch-jumps` 2427 -> 2283. Rich package median active sum moved 2090.4 ms ->
1982.5 ms in that smoke run. `rich-article-fit-batch` became slower with unchanged structure, so
that row should be treated as smoke timing drift rather than evidence for this clamped-path change.

Rich structural patching now has a narrow same-text-node fast path. When both the previous and next
states are clamped cuts in the same source text node, and the target prefix would not be normalized
by the generic trailing-whitespace trim, `patchRich` updates the live `Text.data` directly instead
of removing and cloning the same suffix again. The path is guarded against a root-level ellipsis
occupying the source text node's child index after a previous whitespace-only cut; a focused browser
test covers that boundary. Source Rich counters stayed unchanged after the guard
(`direct-rich-continuous-warm` client reads 2054, `direct-rich-jitter-warm` 1795,
`component-rich-continuous` 314), while same-process package A/B kept client reads fixed and
reduced mutation records: `rich-metadata-affix-batch-continuous` 49824 -> 38544,
`rich-metadata-affix-batch-jitter` 64240 -> 46336,
`rich-inline-markup-batch-continuous` 47296 -> 37648, and
`rich-dense-batch-jumps` 9148 -> 5434.

Rejected experiments:

- Replacing the rich affix clone cache key with `source.isEqualNode(clone)` made component affix
  active time worse, so the retained key is still source `outerHTML` plus geometry signature.
- Flattening the retained Rich probe affix state from `affixes.before` / `affixes.after` into
  top-level fields made the source benchmark slower with unchanged counters, so the grouped affix
  state remains.
- Adding object-identity short-circuiting to `sameBoundaryPoint` did not help Rich source
  benchmarks and made several direct jitter/dense rows slower, so structural point comparison stays
  on offset plus path equality.
- Adding a Rich affix source-identity fast path to avoid `outerHTML` on stable slot wrappers did
  not improve the targeted package `rich-metadata-affix-*` rows and added cache state, so the
  retained cache remains source `outerHTML` plus geometry signature.
- Reordering Rich full-fit probing ahead of layout inspection did not produce a clear win and would
  add an unnecessary fit read to unsupported-overflow cases.
- Replacing suffix child removal with `Range.deleteContents()` dramatically regressed direct Rich
  scenarios, so the simple `lastChild.remove()` loop is the preferred implementation.
- Replacing `lines.some(...)` in `fitsContent` with an explicit helper loop produced mixed text and
  rich benchmark results, including regressions in dense/component rows, so the original compact
  expression remains the clearer implementation.
- A last-line-first variant of that `fitsContent` loop also kept DOM-read counters unchanged but
  made Rich source timing flat to worse, especially component jumps, so it was rejected for the
  same reason.
- Applying the rect-count fit shortcut to the shared Rich path made Rich package/source rows mixed
  to worse. The shortcut remains opt-in from the text clamp helper only.
- Restricting the Rich rect-count shortcut to prepared trees without explicit `br` / `wbr` breaks
  still regressed Rich continuous, jumps, and full-fit warm source rows with unchanged DOM reads, so
  Rich keeps the full shared fit predicate.
- Branching candidate text generation to avoid start/end slice work looked promising in one run but
  regressed against the simpler ASCII-only version when repeated, so `displayTextForKeptCount`
  keeps the straightforward shared prefix/suffix construction.
- Repeating that idea as a narrower `displayTextForKeptCount` branch that avoided slicing the
  unused side for start/end truncation also regressed source timing with unchanged DOM counters, so
  the straight-line candidate builder remains the preferred measured shape.
- Caching Rich boundary point positions, replacing `sharedPath` with a prefix-reuse variant, and
  caching hidden-probe style writes all passed correctness checks but produced mixed or slower Rich
  benchmark rows; none were retained.
- Caching Rich logical runs by prepared source and atomic layout key slightly helped some continuous
  direct rows but regressed dense and component Rich source rows while leaving DOM-read counters
  unchanged, so the direct per-pass run construction remains preferred.
- Replacing Rich trailing `wbr` detection from `tagName.toLowerCase() === "wbr"` to direct
  `nodeName === "WBR"` kept DOM counters unchanged but regressed Rich source timing, so the clearer
  existing tag-name normalization stays.
- An early attempt to remove the multiline shell's `body` ResizeObserver subscription passed browser
  correctness but did not reduce package resize callback counts at all, so it was not retained at
  that stage. After the later entry-size guard excluded body entries from accepted signatures,
  direct body observation became an unused target and was removed.
- Reducing the multiline shell observation set to only the root and affix wrappers also passed the
  focused browser correctness tests, but package smoke showed unchanged resize callback and mutation
  counters. The timing movement also affected unrelated Inline rows, so it was treated as run drift;
  content observation remains part of the layout-change safety net.
- Changing the shared warm-search expansion limit away from two was rejected. A limit of one
  increased Line, Inline, and Rich warm-path reads and timing; a limit of three helped a few Rich
  component rows but regressed Line, Inline, and Rich dense/jump source rows. The existing two-step
  expansion remains the best measured balance for the shared search helper.
- Raising LineClamp's guarded full-fit skip grow window from 24px to 32px did not change source
  benchmark DOM-read counters, including the direct Line warm rows, and only added timing noise.
  The narrower 24px window remains because it already covers the measured small-grow paths without
  broadening the warm-search contract.
- Extending LineClamp's guarded full-fit skip to previous word-boundary fallback results did not
  reduce reads. A new source benchmark row,
  `component-line-word-fallback-continuous`, stayed at 867 client-rect reads with and without the
  fallback-boundary match because correctness requires the skipped full-source probe to reappear as
  the full candidate in the primary word search. The simpler primary-boundary-only guard remains.
- Lazily allocating Rich fallback cut arrays in logical-run construction looked attractive for the
  default grapheme path, but Rich source benchmarks regressed across most rows with unchanged DOM
  counters. The explicit empty fallback array remains the simpler and faster measured shape.
- Accumulating Rich logical-run text cuts directly while walking nodes avoided the temporary
  `currentTextNodes` array, but it did not change any structural counters and made focused package
  Rich smoke mixed to worse (`rich-metadata-affix-batch-continuous` 482.2 ms -> 496.4 ms and
  `rich-inline-markup-batch-continuous` 441.0 ms -> 450.8 ms in the nearby smoke runs, while jitter
  was effectively flat). The original flush shape is easier to read and remains.
- Pre-sizing the ASCII grapheme offset array by setting `.length` before filling looked like a small
  allocation win, but it regressed Text source rows and several Rich source rows with unchanged DOM
  counters. The direct `push` loop remains the measured ASCII fast path.
- Replacing the rest-parameter `combinedSizeSignature` helper with a fixed-parameter /
  `arguments.length` implementation passed checks but worsened Line, Inline, and Rich package smoke
  timing while leaving counters unchanged, so the clearer rest-parameter helper remains.

## Same-process package comparison support

The package benchmark runner now supports `--targets <target-a>,<target-b>[,...]`. Multiple targets

## Inline and Wrap exact-signature follow-up

deduplicated layout work through integer offset-box signatures. That can miss fractional width
changes even when text wrapping or flex wrapping changes at the subpixel boundary.

Another Rich preparation cleanup was rejected: carrying an incremental `childPathKey` through
`buildPreparedRichNodes` instead of calling `path.join(".")` for element keys. It removed one local
join call but added another recursive parameter, and the source Rich benchmark moved mostly slower
with unchanged counters: direct continuous warm `18.6 -> 19.0 ms`, direct jitter warm
`15.3 -> 16.5 ms`, jumps warm `6.1 -> 6.8 ms`, and dense cold `33.4 -> 35.9 ms` in the comparison
run. The existing `pathKey(childPath)` remains shorter and faster.

Top-level text candidate specialization for `ratio <= 0` and `ratio >= 1` was rejected. The source
text benchmark had a small mixed movement, with direct Line continuous slightly faster but Inline
and word-fallback rows flat to slower. The focused package Line CTA rows were a stronger negative
signal with unchanged DOM counters: continuous moved from the nearby `437.7 ms` stable baseline to
`596.5 ms`, and jitter moved from the prior `~530-565 ms` band to `698.2 ms`. The generic
prefix/suffix split remains because it is shorter and package-hotspot timing did not support the
branch specialization.

Skipping the multiline shell's settled root-signature read after recompute was also rejected. Source
component benchmarks looked attractive because `component-line-continuous` bounding-rect reads moved
`422 -> 281` and `component-rich-continuous` moved `332 -> 221`, with client-rect counts unchanged.
The package hotspot rows exposed the hidden cost: Line CTA continuous moved from the retained
`9120` bounding-rect / `5840` client-rect / `12123` mutation-record shape to `7648` / `6640` /
`12923`, and jitter moved from `8960` / `10727` / `17654` to `8640` / `12647` / `19574`. The saved
root read made the settled signature stale enough to trigger extra measurement and DOM churn, so
the post-recompute root refresh remains.

Replacing `point.path.at(-1)` in Rich patch positioning with direct index arithmetic was rejected.
The first source Rich run looked mildly positive with unchanged counters, but the repeat run moved
several hot rows flat to slower, including direct jumps warm around `5.9 -> 8.0 ms` in that local
comparison. With no structural counter movement and worse readability, the clearer `at(-1)` form
remains.

A `maxHeight`-only fit shortcut using `content.getBoundingClientRect()` was also rejected. It passed
browser contracts and converted Line height-card package reads from `1536` bounding-rect / `2043`
client-rect reads to `3579` / `0`, but the total layout-read shape stayed effectively the same and
median active time was neutral (`~106.9 ms` nearby baseline versus `105.9 ms` in the five-sample
experiment). Rich height-card did not benefit because that benchmark also has a line limit. The
shared fit predicate keeps the single `getClientRects()` path because it remains simpler and no
measured scenario justified the extra branch.

A final code-simplifier pass kept only local snapshot cleanups in Line/Rich hot-path guards: the
Line visible-text state and the Line/Rich warm full-fit skip checks now read their nullable state
once before branching. Source Line/Rich benchmark counters stayed unchanged and browser contracts
still passed. A fixed five-sample package rerun after the VS Code CPU spike showed higher active
times on the machine but unchanged structural evidence: Line CTA continuous/jitter remained at
`9120` / `8960` bounding-rect reads, `5840` / `10727` client-rect reads, and `12123` / `17654`
mutation records; Rich metadata continuous/jitter remained at `12320` / `12992` bounding-rect

kept `0` offset-width and offset-height reads. The local active medians were `600.6`, `686.4`,
`583.3`, and `641.9 ms` respectively, while the source helper timings stayed in the normal range,
so active time remains environment-sensitive and DOM counters remain the primary regression signal.

Three follow-up micro-cleanups were measured and rejected. Replacing slot-content
`Array.some(hasSlotContent)` with an explicit loop kept all Line/Rich package counters unchanged
and produced mixed active timings; the loop version was longer and did not justify replacing the
compact expression. Switching Rich text-node reads from `textContent` to `Text.data` also kept
source counters unchanged but did not produce a stable timing win when repeated in the same
environment. Finally, cloning rich text nodes with `document.createTextNode(data)` instead of
`cloneNode(true)` failed to reduce the tracked clone counter and slowed most source Rich rows. The
existing slot filter and rich text patching code remain the simpler measured shape.
