# Vue Clamp 1.4.0 to 1.5.0 performance analysis

## Data source

This note is based on the package benchmark matrix generated from `/tmp/vue-clamp-matrix-2s-cv`
and stored locally as `journey/research/310-package-benchmark-matrix.local.json`.

The benchmark compares published package versions through the public component API. The report
sampling mode uses at least 2 seconds of measured wall time per scenario, a minimum of 3 measured
samples, median active time as the primary timing value, and active-time CV as the low-confidence
marker.

## Aggregate result

| Component     | 1.4.0 active | 1.5.0 active |  Delta | Rect reads | Item slot calls | Interpretation                                                              |
| ------------- | -----------: | -----------: | -----: | ---------: | --------------: | --------------------------------------------------------------------------- |
| LineClamp     |    2094.1 ms |    2225.7 ms |  +6.3% |       0.0% |             N/A | Timing-only regression in the same measured work shape.                     |
| InlineClamp   |    1796.6 ms |    1675.3 ms |  -6.7% |       0.0% |             N/A | Runtime/render-path improvement, not fewer DOM reads.                       |
| RichLineClamp |    2168.4 ms |    2185.5 ms |  +0.8% |       0.0% |             N/A | Mostly neutral; small timing-only movement.                                 |
| WrapClamp     |    8026.3 ms |    3886.3 ms | -51.6% |     -60.4% |          -70.0% | Dominant improvement from less DOM probing and fewer item slot invocations. |

Across all comparable scenarios, 1.5.0 improves active time by 29.2%, settled time by 9.1%, rect
reads by 58.0%, item slot calls by 70.0%, and long-task count by 17.5%.

## Main improvements

The 1.5.0 improvement is dominated by WrapClamp. The largest scenario wins are all grow-heavy
wrapped-item cases:

| Scenario                                | Active delta | Rect reads | Item slot calls |
| --------------------------------------- | -----------: | ---------: | --------------: |
| `wrap-no-affix-tiny-item-wide-grow`     |       -80.0% |     -87.4% |          -92.5% |
| `wrap-no-affix-heavy-item-grow`         |       -66.8% |     -67.5% |          -72.2% |
| `wrap-no-affix-hidden-grow`             |       -65.9% |     -67.5% |          -72.2% |
| `wrap-static-before-dynamic-after-grow` |       -62.8% |     -71.8% |          -63.6% |
| `wrap-static-after-grow`                |       -63.3% |     -77.5% |          -68.0% |
| `wrap-before-affix-grow`                |       -55.9% |     -43.4% |          -78.6% |

The source-level reason is the WrapClamp solver rewrite in 1.5.0:

- It records item widths during real measurements and reuses those widths for later estimates.
- It can estimate no-affix / static-flow counts without rendering every candidate item.
- Grow paths materialize a bounded frontier and then toggle existing item shells during search
  instead of repeatedly re-rendering each candidate prefix.
- The fallback linear settlement still exists, but it is reached after cheaper proof/hint paths.

The item slot call reduction is especially important because the benchmark intentionally renders
many WrapClamp rows with public `item` slots. In 1.4.0, grow paths repeatedly invoked item slots for
candidate prefixes. In 1.5.0, the renderer only renders up to the visible count or bounded
materialized count, so heavy item content no longer scales with every probe.

InlineClamp also improves in continuous width-sweep scenarios:

| Scenario                              | Active delta | Work counters |
| ------------------------------------- | -----------: | ------------- |
| `inline-path-middle-batch-continuous` |       -17.3% | unchanged     |
| `inline-path-end-batch-continuous`    |       -17.0% | unchanged     |

Because DOM-read counters are unchanged, this is best attributed to component-shell/render-path
changes from the SFC migration and local state simplification, not to a changed text-clamp search
algorithm.

## Main regressions

The largest timing regressions are:

| Scenario                               | Active delta | Absolute delta | Work counters | Confidence      |
| -------------------------------------- | -----------: | -------------: | ------------- | --------------- |
| `line-summary-batch-continuous`        |       +44.6% |       +50.9 ms | unchanged     | CV 4.2% -> 0.7% |
| `line-cta-affix-batch-continuous`      |        +9.0% |       +46.8 ms | unchanged     | CV 2.5% -> 2.3% |
| `rich-metadata-affix-batch-continuous` |        +4.6% |       +24.5 ms | unchanged     | CV 1.3% -> 1.9% |
| `inline-path-middle-batch-jitter`      |        +5.2% |       +23.2 ms | unchanged     | CV 7.5% -> 1.6% |
| `rich-metadata-affix-batch-jitter`     |        +3.1% |       +19.2 ms | unchanged     | CV 0.3% -> 1.4% |
| `line-prefixed-summary-batch-jumps`    |       +25.5% |       +11.5 ms | unchanged     | CV 1.8% -> 7.6% |
| `inline-split-file-path-batch-jumps`   |       +13.9% |        +8.2 ms | unchanged     | CV 5.1% -> 3.4% |

For LineClamp, the most important regression is `line-summary-batch-continuous`: it is a native
eligible multiline default-ellipsis row with no affix. The native implementation path and measured
DOM-read counts did not change between 1.4.0 and 1.5.0. The source-level difference is the component
shell: 1.5.0 moved LineClamp from a hand-written render function to an SFC/template shell and
`defineModel`-based controlled state. A concrete likely contributor is that the SFC template renders
the reusable `MultilineAffixSlot` functional component for both before and after positions even when
there is no slot content. In 1.4.0 the hand-written render function skipped those wrappers when slot
content was absent.

For the affix LineClamp and RichLineClamp rows, rect reads, resize callbacks, mutation records, and
component instance counts are unchanged. These regressions are therefore not evidence of extra
clamp probes. They are timing-only overhead in the same externally observable workload, most likely
from the SFC shell, the reusable affix-slot functional component path, and reactive style/render
boundaries.

Small WrapClamp regressions such as `wrap-table-demo-width-sweep` (+3.4%) and
`wrap-after-affix-shrink` (+2.6%) also have unchanged rect and slot counters. They are not the main
story and should not be treated as solver regressions unless a profiler shows new hot frames.

## Bottom line

1.5.0 is a clear WrapClamp performance release. The solver rewrite cuts the expensive work that
actually matters for wrapped-item scenarios: DOM rect probing and public item-slot invocation.

The regressions are real in the timing data, especially `line-summary-batch-continuous`, but they
are not caused by extra DOM measurement. They are component-shell/runtime overhead introduced while
migrating to SFCs and simplifying public typing. If those regressions matter enough to optimize, the
next diagnostic step should be an isolated commit-level run or browser profile focused on the
native LineClamp SFC path and affix slot render path.

## DefineRender follow-up experiment

After this analysis, `LineClamp`, `RichLineClamp`, and `InlineClamp` were migrated from authored
templates to setup-local render functions bound with `defineRender(render)`. `WrapClamp` was already
render-only. The goal was to test whether avoiding template-emitted `MultilineAffixSlot` component
calls would recover 1.5.0 timing regressions.

The package benchmark was rerun against the current workspace and published `vue-clamp@1.5.0` using
the same public matrix and report sampling mode:

| Component     | Published 1.5.0 | Current defineRender | Delta |
| ------------- | --------------: | -------------------: | ----: |
| All           |       9903.3 ms |            9975.5 ms | +0.7% |
| LineClamp     |       2230.9 ms |            2160.1 ms | -3.2% |
| InlineClamp   |       1715.6 ms |            1819.0 ms | +6.0% |
| RichLineClamp |       2162.0 ms |            2147.9 ms | -0.6% |
| WrapClamp     |       3794.8 ms |            3848.4 ms | +1.4% |

The intended Line/Rich recovery did show up:

| Scenario                               | Published 1.5.0 | Current defineRender |  Delta |
| -------------------------------------- | --------------: | -------------------: | -----: |
| `line-summary-batch-continuous`        |        166.3 ms |             142.0 ms | -14.6% |
| `line-cta-affix-batch-continuous`      |        566.8 ms |             544.4 ms |  -4.0% |
| `line-cta-affix-batch-jitter`          |        844.3 ms |             811.9 ms |  -3.8% |
| `rich-metadata-affix-batch-continuous` |        551.4 ms |             536.5 ms |  -2.7% |
| `rich-metadata-affix-batch-jitter`     |        640.4 ms |             627.8 ms |  -2.0% |

But migrating InlineClamp to a manual render function was not a win in this run:

| Scenario                              | Published 1.5.0 | Current defineRender |  Delta |
| ------------------------------------- | --------------: | -------------------: | -----: |
| `inline-path-middle-batch-jitter`     |        418.8 ms |             456.8 ms |  +9.1% |
| `inline-path-middle-batch-continuous` |        416.9 ms |             435.8 ms |  +4.5% |
| `inline-split-file-path-batch-jumps`  |         39.7 ms |              60.4 ms | +52.1% |

Counters were effectively unchanged in the defineRender comparison, so the movement is still
render/runtime timing rather than changed clamp probes. The practical takeaway is that render-only
SFCs are promising for LineClamp and mildly helpful for RichLineClamp, but should not automatically
be assumed faster for InlineClamp. If optimizing purely for current benchmark totals, keep the
LineClamp/RichLineClamp render-only migration and reconsider InlineClamp separately.

The deeper reason is that the experiment changes two different costs in opposite directions:

- `LineClamp` and `RichLineClamp` templates in published 1.5.0 always instantiate the reusable
  `MultilineAffixSlot` VNode path for both before and after positions. In no-affix rows this is
  pure shell work; in affix rows it adds a component boundary and prop patching around slot
  rendering. The render-only experiment removes or inlines that boundary, which recovers the
  targeted Line/Rich regressions.
- Authored templates also receive Vue compiler output: block tracking, patch flags, hoisted dynamic
  prop key arrays, and generated template-ref handling. Handwritten `h()` render functions do not
  get those compiler optimizations automatically. `InlineClamp` had no `MultilineAffixSlot` boundary
  to remove, so the experiment mostly traded a compiler-optimized template for fresh hand-built
  VNodes, props objects, children arrays, and function refs. That is why InlineClamp regressed even
  though its DOM-read and mutation counters were unchanged.
- The LineClamp recovery is partial because the 1.4.0 render path was not just render-only; it was a
  leaner hand-written `defineComponent` path. The current SFC path still carries `defineModel` /
  `useModel`, template-oriented computed boundaries such as `rootStyle`, `contentStyle`,
  `nativeMode`, `bodyStyle`, and `renderedText`, plus function refs for DOM nodes. The experiment
  removed the biggest avoidable template component boundary, but it did not recreate the entire
  1.4.0 runtime shape.

Most of the measured movement is in `updateMs`, not `settleWaitMs`. Counters for layout reads,
resize callbacks, and mutation records are nearly unchanged, so these deltas should be interpreted
as Vue render/patch/ref/slot overhead rather than browser layout or clamp-search work.

## Hybrid SFC render optimization pass

The follow-up implementation kept the public SFC shell constraint but stopped treating render-only
as universally faster:

- `InlineClamp` returned to an authored template with `useTemplateRef` because its runtime structure
  is static and the template compiler's block/patch/ref output consistently fits that shape better
  than a handwritten `h()` render function.
- `LineClamp` and `RichLineClamp` stayed render-only, but root/content/body/text/probe refs now use
  direct shallow ref objects for stable element refs. Function refs remain only for affix wrappers,
  where appearance/disappearance must schedule a reclamp after Vue commits slot DOM.
- One-use render aliases such as native mode, body style, rendered text, and slot props were moved
  out of `computed` effects and into render-local values, keeping `computed` for preparation work
  that is actually cached across reclamps.
- `LineClamp` also reuses stable style objects on the common no-`maxHeight` path and avoids rendering
  a null source-text placeholder when the visible text is not accessibility-rewritten.

The first full matrix after the hybrid pass showed the expected direction versus the previous
all-`defineRender` workspace run:

| Component     | All-defineRender current | Hybrid pass |  Delta |
| ------------- | -----------------------: | ----------: | -----: |
| All           |                9975.5 ms |   9091.9 ms |  -8.9% |
| LineClamp     |                2160.1 ms |   2071.6 ms |  -4.1% |
| InlineClamp   |                1819.0 ms |   1360.0 ms | -25.2% |
| RichLineClamp |                2147.9 ms |   1911.3 ms | -11.0% |
| WrapClamp     |                3848.4 ms |   3749.0 ms |  -2.6% |

Two repeated runs of the final code showed meaningful cross-run drift, especially in long
Inline/Rich continuous scenarios, so the conservative reading uses their average rather than a
single best run:

| Component     | All-defineRender current | Final repeated avg | Delta |
| ------------- | -----------------------: | -----------------: | ----: |
| All           |                9975.5 ms |          9530.1 ms | -4.5% |
| LineClamp     |                2160.1 ms |          2060.9 ms | -4.6% |
| InlineClamp   |                1819.0 ms |          1652.9 ms | -9.1% |
| RichLineClamp |                2147.9 ms |          2168.4 ms | +0.9% |
| WrapClamp     |                3848.4 ms |          3647.9 ms | -5.2% |

Against the published historical per-scenario best matrix, the repeated final-code average is close
to the goal but not uniformly best in every individual row:

| Component     | Historical best sum | Final repeated avg |   Gap |
| ------------- | ------------------: | -----------------: | ----: |
| All           |           9676.5 ms |          9530.1 ms | -1.5% |
| LineClamp     |           2075.5 ms |          2060.9 ms | -0.7% |
| InlineClamp   |           1628.9 ms |          1652.9 ms | +1.5% |
| RichLineClamp |           2126.5 ms |          2168.4 ms | +2.0% |
| WrapClamp     |           3845.6 ms |          3647.9 ms | -5.1% |

The remaining positive gaps are concentrated in a small set of rows, mostly within the observed
cross-run drift band: `inline-path-middle-batch-continuous`, `rich-inline-markup-batch-continuous`,
several Rich jump rows, and the small `line-title-single` row. The code-level conclusion is that the
clean hybrid shell is worth keeping, while further claims about exact per-scenario historical-best
parity need either same-process target comparison or a benchmark runner that alternates targets in
one browser session to reduce drift.
